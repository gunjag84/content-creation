# Phase 03.1: Visual Slide Editor - Research

**Researched:** 2026-03-17
**Domain:** React/Zustand state management, per-slide overrides, undo/redo, preset persistence
**Confidence:** HIGH

## Summary

Phase 03.1 extends the existing Step 3 editor (two-panel, 40/60 layout) with per-slide visual controls that override the template's zone definitions. The template system already defines zones with absolute pixel positions, font sizes, and roles — what's missing is per-slide override capability so each slide can shift a zone's position or change font weight/color without touching the template itself.

The key architectural insight: per-slide overrides live on the `Slide` type (in `useCreatePostStore`) as an optional `zone_overrides` map keyed by zone ID. The render pipeline in `Step4RenderReview.buildSlideHTML()` reads zones from `template.zones_config`, then merges any per-slide overrides before building the HTML. This keeps templates immutable while allowing per-slide customization. No new IPC handlers are needed until DB persistence of overrides is required; the wizard session holds the state.

Undo/redo should be implemented as a history stack inside the Zustand wizard store, not as a separate library. The store already manages `generatedSlides` as an array — wrapping mutations with push-to-history / pop-from-history is straightforward and avoids adding a dependency. Presets are saved to JSON via the existing `settingsService` pattern (or a new separate presets JSON file) and surfaced in a dropdown in Step 3.

**Primary recommendation:** Add `zone_overrides` to the `Slide` type, wire per-slide controls into Step 3's left panel under the existing Overlay Opacity section, store history in the wizard Zustand store, and persist presets to a JSON file using the same pattern as settings.

### Pattern 1: Per-Slide Zone Override Data Model

**What:** Each slide carries an optional `zone_overrides` map: `{ [zoneId: string]: Partial<ZoneOverride> }`. Override fields are applied on top of the template zone definition at render time.

**When to use:** Every time a slide-specific visual tweak is needed (position shift, font size change, color change, font weight change).

**ZoneOverride type (add to `src/shared/types/generation.ts`):**

```typescript
// Per-slide override for a single zone
export interface ZoneOverride {
  x?: number          // absolute pixel offset from template default
  y?: number
  width?: number
  height?: number
  fontSize?: number   // override zone.fontSize
  fontWeight?: 'normal' | 'bold' | '600' | '700'
  color?: string      // override brand color for this zone on this slide
}

// Extend existing Slide type
export interface Slide {
  // ... existing fields ...
  zone_overrides?: Record<string, ZoneOverride>   // keyed by zone.id
}
```

**Merge logic in `buildSlideHTML()` (Step4RenderReview.tsx):**

```typescript
const effectiveZone = {
  ...zone,
  ...(slide.zone_overrides?.[zone.id] ?? {})
}
// use effectiveZone.x, effectiveZone.fontSize, etc.
```

The same merge logic applies in `LivePreview` for Step 3 instant preview.

### Pattern 2: Undo/Redo via History Stack in Zustand

**What:** History is a stack of `Slide[]` snapshots stored in the wizard store. Before any mutating action, the current slides array is pushed to the stack. Undo pops the stack. Redo uses a forward stack.

**When to use:** All slide mutations (text edits, zone overrides, overlay opacity changes).

**State additions to `useCreatePostStore`:**

```typescript
// History
slideHistory: Slide[][]       // past states, index 0 = oldest
slideHistoryFuture: Slide[][]  // states after undo

// Actions
pushHistory: () => void        // call BEFORE mutating generatedSlides
undo: () => void
redo: () => void
canUndo: () => boolean
canRedo: () => boolean
```

**Implementation pattern:**

```typescript
pushHistory: () => set((state) => ({
  slideHistory: [...state.slideHistory, state.generatedSlides].slice(-50), // cap at 50
  slideHistoryFuture: []  // clear redo on new action
})),

undo: () => set((state) => {
  if (state.slideHistory.length === 0) return state
  const previous = state.slideHistory[state.slideHistory.length - 1]
  return {
    generatedSlides: previous,
    slideHistory: state.slideHistory.slice(0, -1),
    slideHistoryFuture: [state.generatedSlides, ...state.slideHistoryFuture].slice(0, 50)
  }
}),

redo: () => set((state) => {
  if (state.slideHistoryFuture.length === 0) return state
  const next = state.slideHistoryFuture[0]
  return {
    generatedSlides: next,
    slideHistory: [...state.slideHistory, state.generatedSlides],
    slideHistoryFuture: state.slideHistoryFuture.slice(1)
  }
})
```

**Critical:** `pushHistory()` must be called by the store action, not the component. Update `setSlide()` and any new zone override actions to call `pushHistory()` internally before mutating.

**Cap history at 50 entries** to prevent unbounded memory growth during long sessions.

### Pattern 3: Preset Persistence

**What:** Presets save a named `ZoneOverride` configuration (a map of zoneId overrides + their associated `overlay_opacity`) that can be applied to any slide. Stored in a dedicated `presets.json` in `app.getPath('userData')`.

**When to use:** When a user wants to reuse a visual configuration (e.g., "emphasize hook" = hook font bold + larger, body slightly smaller).

**Preset type:**

```typescript
export interface SlidePreset {
  id: string              // nanoid or crypto.randomUUID()
  name: string
  zone_overrides: Record<string, ZoneOverride>
  overlay_opacity?: number
  created_at: number      // Unix timestamp
}
```

**Preset service pattern (mirrors `settings-service.ts`):**

### Pattern 4: SlideZoneOverrides Component in Step 3

**What:** A collapsible panel in Step 3's left column (below the existing Overlay Opacity slider) that shows zone-specific controls for the active slide.

**UI structure:**

```
[ Zone Overrides ] ← collapsible section header

For each zone in template.zones_config:
  Zone: "Hook" (blue badge)
    Font size:  [slider 12-120]   [value px]
    Font weight: [Normal] [Bold]
    Color:      [color swatch → picker]
    Position X: [slider or number input, ±100px range]
    Position Y: [slider or number input, ±100px range]

  Zone: "Body" (green badge)
    [same controls]

[ Apply Preset ▼ ] [ Save as Preset ]
```

**Key UX decisions:**
- Position sliders show delta from template position (±100px), not absolute values. Label: "X offset" not "X position". This avoids confusion with the underlying template.
- Color picker uses existing `react-colorful` (already used in `BrandColorPicker`).
- Changes trigger `pushHistory()` then update `zone_overrides` on the active slide.
- Changes in Step 3 update `LivePreview` immediately (LivePreview must be upgraded to render using the same `buildSlideHTML` logic or an equivalent CSS-based render).

### Pattern 5: LivePreview Upgrade

**Current state:** `LivePreview` shows a simple branded card (text only, no real template layout). It does NOT reflect zone positions.

**Needed:** `LivePreview` should render actual zone positions including overrides, so Step 3 changes are immediately visible.

**Approach:** Lift `buildSlideHTML` out of `Step4RenderReview` into a shared utility (`src/renderer/src/lib/buildSlideHTML.ts`) that both `LivePreview` and `Step4RenderReview` call. `LivePreview` renders the HTML in an iframe or a scaled div (same `scale(0.3)` transform pattern it already uses for `templateHtml`).

The `LivePreview` component already accepts `templateHtml` prop and has the scaled-div rendering path. The only change needed: pass a computed `templateHtml` from the parent (Step 3) instead of passing the raw slide.

**Anti-Patterns to Avoid**

- **Duplicating `buildSlideHTML` logic:** It already exists in `Step4RenderReview`. Extract to shared utility before adding override support. Keeping two copies guarantees drift.
- **Storing overrides in the DB schema in Phase 03.1:** The slide DB schema (`slides` table) has no `zone_overrides` column. Adding a DB column requires a migration. For Phase 03.1, keep overrides in wizard session state only (lost on navigation away, which is acceptable for the current phase). DB persistence is a future concern.
- **One history entry per keystroke:** Debounce text field changes (already partially done with the existing approach). Only push history on blur/commit for text edits, not on every character. Zone override sliders can push on `onValueCommit` (Radix Slider fires this on mouse-up).
- **Using an external undo library (immer-history, use-undo, etc.):** The store is simple enough. Adding a dependency to solve a 30-line problem is over-engineering.

## Common Pitfalls

### Pitfall 1: LivePreview Disconnect

**What goes wrong:** Zone overrides update the Zustand store, but LivePreview still shows the simple text card. User sees no visual feedback until Step 4 render.

**Why it happens:** `LivePreview` currently does not read `template.zones_config` or `zone_overrides`. It was a placeholder that was never upgraded.

**How to avoid:** Extract `buildSlideHTML` to a shared utility first (Wave 1 task). Then pass computed HTML to `LivePreview` in Step 3. `LivePreview` already has the `templateHtml` prop path — just use it.

**Warning signs:** If Step 3 tests render `LivePreview` without a template, the simple card path still runs. Verify the template loading in Step 3 is triggered before the visual editor renders.

### Pitfall 2: Template Not Loaded in Step 3

**What goes wrong:** Step 3 currently receives slides from the wizard store but does NOT load the template. Template loading happens in Step 4 (`useEffect` calls `window.api.templates.list()`). Zone override controls need the zone list, which comes from the template.

**Why it happens:** Step 3 was designed as a text editor, not a visual editor. Template loading was deferred to Step 4.

**How to avoid:** Load the template in Step 3 (or in Step 1 when user selects a template, storing `selectedTemplateId` in the wizard store). The wizard store already has `selectedPillar`, `selectedTheme` etc. — add `selectedTemplateId` and load template zones on Step 3 mount.

**Warning signs:** `zones_config` parsing yields empty array → zone controls panel shows nothing → user cannot override anything.

### Pitfall 3: History Stack Growing Without Bound

**What goes wrong:** Every slider `onValueChange` fires 20+ times during a drag. Each fires pushHistory. Stack grows to 1000 entries within seconds.

**Why it happens:** Radix Slider `onValueChange` fires continuously. `onValueCommit` only fires on mouse-up.

**How to avoid:** Use `onValueCommit` for history pushes on sliders. Use `onBlur` for text inputs. Only call `pushHistory` once per logical user action, not per intermediate event.

**Warning signs:** Memory usage climbing during slider interaction. Undo requires 50+ presses to go back one logical change.

### Pitfall 4: Zone ID Mismatch Between Template and Overrides

**What goes wrong:** Template zones have IDs like `zone-1710000000-abc123`. If template is re-saved with new zones (different IDs), all stored overrides become orphaned.

**Why it happens:** Zone IDs are generated at creation time with `Date.now()`. If a user edits the template and re-saves, zones get new IDs.

**How to avoid:** For Phase 03.1, this is acceptable. Overrides are session-only (not persisted to DB). Presets should store the zone type (`hook`/`body`/`cta`) as a fallback lookup key alongside the zone ID, so when IDs differ, the preset can still find the right zone by type.

**Warning signs:** After template edit, applying a saved preset has no visible effect.

### Pitfall 5: buildSlideHTML Lives in a Component File

**What goes wrong:** `buildSlideHTML` is currently a private function inside `Step4RenderReview.tsx`. Sharing it requires extracting it.

**Why it happens:** It was written as a local function during Phase 3 development.

**How to avoid:** Move to `src/renderer/src/lib/buildSlideHTML.ts` in the first wave. This is a prerequisite for LivePreview upgrade and zone override integration.

## Code Examples

### Merging Zone Overrides at Render Time

```typescript
// src/renderer/src/lib/buildSlideHTML.ts (extracted + extended)
function buildZoneElement(zone: Zone, slide: Slide, settings: Settings): string {
  const override = slide.zone_overrides?.[zone.id] ?? {}
  const effectiveZone = { ...zone, ...override }

  const x = effectiveZone.x
  const y = effectiveZone.y
  const width = effectiveZone.width
  const height = effectiveZone.height
  const fontSize = effectiveZone.fontSize ?? zone.fontSize

  // font weight and color from override, else from brand guidance defaults
  const fontWeight = override.fontWeight ?? 'normal'
  const color = override.color ?? (zone.type === 'body' ? secondaryColor : primaryColor)

  return `<div style="position:absolute;left:${x}px;top:${y}px;width:${width}px;height:${height}px;font-size:${fontSize}px;color:${color};font-weight:${fontWeight};...">${text}</div>`
}
```

### Zustand Store: pushHistory Pattern

```typescript
// In useCreatePostStore.ts
// Mutating action that supports undo
setZoneOverride: (slideIndex: number, zoneId: string, override: Partial<ZoneOverride>) =>
  set((state) => {
    const slides = [...state.generatedSlides]
    const slide = { ...slides[slideIndex] }
    slide.zone_overrides = {
      ...(slide.zone_overrides ?? {}),
      [zoneId]: { ...(slide.zone_overrides?.[zoneId] ?? {}), ...override }
    }
    slides[slideIndex] = slide
    // Push history before mutation (capture old state)
    const newHistory = [...state.slideHistory, state.generatedSlides].slice(-50)
    return {
      generatedSlides: slides,
      slideHistory: newHistory,
      slideHistoryFuture: []
    }
  })
```

### Preset Application Pattern

```typescript
// Apply a preset to the active slide
applyPreset: (slideIndex: number, preset: SlidePreset) =>
  set((state) => {
    const slides = [...state.generatedSlides]
    const slide = { ...slides[slideIndex] }
    slide.zone_overrides = preset.zone_overrides
    if (preset.overlay_opacity !== undefined) {
      slide.overlay_opacity = preset.overlay_opacity
    }
    slides[slideIndex] = slide
    const newHistory = [...state.slideHistory, state.generatedSlides].slice(-50)
    return {
      generatedSlides: slides,
      slideHistory: newHistory,
      slideHistoryFuture: []
    }
  })
```

## Open Questions

1. **Where does the user select which template is active during the wizard?**
   - What we know: Step 4 loads `templates[0]` as a hardcoded placeholder (`// assume first template for now`).
   - What's unclear: Phase 03.1 zone controls require knowing the template zones. Should this phase also fix template selection in Step 1, or load template in Step 3 by index?
   - Recommendation: Add `selectedTemplateId` to the wizard store and load it in Step 3 on mount. Keep template selection in Step 1 as a low-priority separate concern (can default to first template if only one exists, which covers the single-user beta case).

2. **Should zone position overrides be relative (delta from template) or absolute?**
   - What we know: Template zones store absolute pixel positions at 1080px width.
   - What's unclear: Relative (delta) is safer because it survives template re-positioning. Absolute is simpler to implement.
   - Recommendation: Store absolute values in `ZoneOverride` (same coordinate space as Zone). Show as delta in the UI (subtract template baseline, display ±N). This gives simple storage with intuitive display.

3. **Preset storage format — should presets be per-template or global?**
   - What we know: Zone IDs are template-specific.
   - What's unclear: If presets store zone IDs, applying a preset designed for Template A to a slide using Template B is meaningless.
   - Recommendation: Tag each preset with `template_id`. When loading presets, filter to those matching the current template (with a fallback "type-based" match for presets created before template ID tagging). Phase 03.1 can start with global presets and fallback to zone type matching.

## Data Flow Summary

```
Template (zones_config JSON)
    ↓ parsed in Step 3 mount
Zone[] array
    ↓ displayed in SlideZoneOverrides panel
User adjusts zone X offset / font size / color
    ↓ pushHistory() + setZoneOverride(slideIndex, zoneId, override)
Slide.zone_overrides updated in Zustand store
    ↓ passed to buildSlideHTML(slide, template, settings)
Effective zone = merge(templateZone, slide.zone_overrides[zoneId])
    ↓
HTML string with correct position/font/color
    ↓
LivePreview (Step 3): scaled div / iframe
Step 4: BrowserWindow.capturePage() → PNG
```
