  **Fix A: Background image loading (file:// protocol blocked)**

  The renderer process with `contextIsolation: true` and `sandbox: false` may still block `file://` URLs depending on Electron version. The pattern established in Phase 1 is to use base64 data URLs.

  Solution: Create an IPC handler that reads an image file and returns a base64 data URL:

  In `src/main/ipc/fonts.ts` (or a new `src/main/ipc/files.ts`):
  ```typescript
  ipcMain.handle('file:read-as-data-url', async (_event, filePath: string) => {
    const data = await fs.readFile(filePath)
    const ext = path.extname(filePath).toLowerCase().replace('.', '')
    const mimeType = ext === 'jpg' ? 'image/jpeg' : ext === 'png' ? 'image/png' : `image/${ext}`
    return `data:${mimeType};base64,${data.toString('base64')}`
  })
  ```

  In preload/types.ts add: `readFileAsDataUrl: (path: string) => Promise<string>`
  In preload/index.ts add: `readFileAsDataUrl: (path) => ipcRenderer.invoke('file:read-as-data-url', path)`

  In TemplateBuilder.tsx, replace `loadBackgroundImage`:
  ```typescript
  const loadBackgroundImage = async (imagePath: string) => {
    try {
      const dataUrl = await window.api.readFileAsDataUrl(imagePath)
      const img = new Image()
      img.onload = () => setBackgroundImage(img)
      img.onerror = () => setBackgroundImage(null)
      img.src = dataUrl
    } catch (err) {
      console.error('Failed to load background image:', err)
      setBackgroundImage(null)
    }
  }
  ```

  **Fix B: Canvas sizing - too large for the window**

  In ZoneEditor.tsx, the canvas tries to fill its container width. But the container may be taking full page width without constraints. Fix:

  1. Set a maximum width on the canvas container: `max-w-3xl` (768px) or a fixed maxWidth that fits well in the settings content area
  2. The aspect ratio math is correct (width/CANVAS_WIDTH * canvasHeight) but the initial `containerSize` of `{ width: 700, height: 875 }` may be overwritten by ResizeObserver to a very large value if the container is unconstrained
  3. Add `max-w-[700px] mx-auto` to the canvas container div to constrain it
  4. The canvas should show at a reasonable preview size (roughly 500-700px wide) with the full 1080-scale coordinates mapped to this display size

  **Fix C: Zone drawing not persisting**

  Debug the zone drawing flow. The issue may be in the `handleStageMouseUp` handler:
  - The `tempZone` check: `if (isDrawing && tempZone && drawStart)` - if any of these are false, the zone won't be committed
  - The min size check: `tempZone.width >= MIN_ZONE_SIZE && tempZone.height >= MIN_ZONE_SIZE` where MIN_ZONE_SIZE=50. At display scale, a 50px drag might map to much more than 50 in canvas coordinates, or vice versa. The issue is that `tempZone` stores coordinates in 1080-scale but the MIN_ZONE_SIZE check may be comparing against display-scale values
  - Check if `handleStageMouseMove` properly calculates scaled coordinates. Currently: `const x = pos.x / scale` and `const y = pos.y / scale` which converts display coords to canvas coords. Then `width = Math.abs(x - drawStart.x)` is in canvas coords. MIN_ZONE_SIZE=50 in canvas coords means 50/1080*containerWidth pixels on screen - about 32-45px, which should be draggable
  - The more likely issue: `handleStageMouseUp` has `if (isDrawing && tempZone && drawStart)` but `handleStageMouseMove` only runs if `isDrawing && drawStart`. If mouse moves too fast, `handleStageMouseMove` might not fire, leaving `tempZone` null. Add a fallback: if `tempZone` is null at mouseUp but `drawStart` exists, calculate the zone from drawStart to current pointer position
  - Also: After committing, `setDrawMode(false)` immediately disables draw mode. This might cause the user to think zones aren't persisting because they have to click "Draw Zone" again for each zone. Consider keeping draw mode active until explicitly cancelled

  <action>
  **Root cause:** The brand preview calls `window.api.renderToPNG()` which returns a base64-encoded PNG data URL. This large string (potentially hundreds of KB) gets set as `img.src`. During development, Vite's HMR/websocket connections or subsequent fetch requests may include this data in headers (referer, cookie), causing the Vite dev server to reject the request with HTTP 431 Request Header Fields Too Large.

  **Fix approach - prevent the large data URL from leaking into request headers:**

  1. Instead of setting the base64 data URL directly on an `<img>` element, use a Blob URL:
     ```typescript
     const renderPreview = async () => {
       setIsRendering(true)
       try {
         const html = generatePreviewHTML(latestGuidance.current)
         const dataUrl = await window.api.renderToPNG(html, { width: 1080, height: 1350 })

         // Convert data URL to Blob URL to avoid header pollution
         const response = await fetch(dataUrl)
         const blob = await response.blob()

         // Revoke previous blob URL to prevent memory leaks
         if (previewBlobUrl) {
           URL.revokeObjectURL(previewBlobUrl)
         }

         const blobUrl = URL.createObjectURL(blob)
         setPreviewBlobUrl(blobUrl)
       } catch (error) {
         console.error('Preview render failed:', error)
       } finally {
         setIsRendering(false)
       }
     }
     ```

  2. Add a `previewBlobUrl` state variable and display it:
     ```typescript
     const [previewBlobUrl, setPreviewBlobUrl] = useState<string | null>(null)
     ```

  3. Clean up blob URL on unmount:
     ```typescript
     useEffect(() => {
       return () => {
         if (previewBlobUrl) URL.revokeObjectURL(previewBlobUrl)
       }
     }, [previewBlobUrl])
     ```

  4. Use `previewBlobUrl` in the img src instead of `previewDataUrl`

  5. If `fetch(dataUrl)` doesn't work in Electron renderer, use an alternative conversion:
     ```typescript
     function dataUrlToBlob(dataUrl: string): Blob {
       const parts = dataUrl.split(',')
       const mime = parts[0].match(/:(.*?);/)?.[1] || 'image/png'
       const raw = atob(parts[1])
       const arr = new Uint8Array(raw.length)
       for (let i = 0; i < raw.length; i++) arr[i] = raw.charCodeAt(i)
       return new Blob([arr], { type: mime })
     }
     ```

  The BrandPreview container uses light theme classes: `bg-gray-50 border-gray-200`, `text-gray-500`, `text-gray-700`. Update to dark theme: `bg-slate-800 border-slate-700`, `text-slate-400`, `text-slate-200`.

But since the preview is rendered by the main process RenderService (Puppeteer), file:// URLs may work there since it's not subject to renderer security restrictions.
