<task type="auto">
  <name>Task 1: Fix better-sqlite3 rebuild and Zod validation</name>
  <files>package.json, src/shared/types/settings.ts, src/main/services/settings-service.ts</files>
  <action>
  **better-sqlite3 rebuild:**
  The native module was compiled against a different Node.js version than Electron uses. Fix by running `npx electron-rebuild` (or `npx @electron/rebuild`) to recompile better-sqlite3 for the current Electron version. If electron-rebuild is not in devDependencies, add it first with `npm install -D @electron/rebuild`. Then run the rebuild. Also ensure package.json has a postinstall script: `"postinstall": "electron-rebuild"` (or the electron-vite equivalent - check if electron-vite already handles this via its build config).

  Note: electron-vite may handle native module rebuilding via its build config. Check electron.vite.config.ts for externals configuration. If `better-sqlite3` is already in externals, the rebuild step may just need to be run manually once. If electron-vite has a `buildPlugin` or `nativeModulePlugin`, leverage that.

  **Zod validation fix:**
  In `src/main/services/settings-service.ts`, the `load()` method does a shallow merge: `{ ...DEFAULT_SETTINGS, ...parsed }`. This fails when a previously saved settings.json has fields set to `undefined` or is missing nested fields. Fix by implementing a deep merge that only overrides defaults with defined values:

  ```typescript
  // Replace shallow merge with deep merge
  const merged = deepMergeSettings(DEFAULT_SETTINGS, parsed)
  ```

  Create a `deepMergeSettings` function in settings-service.ts that:
  1. Iterates over DEFAULT_SETTINGS keys
  2. For each key: if parsed[key] is undefined or null, use DEFAULT_SETTINGS[key]
  3. If parsed[key] is an object (not array), recursively merge
  4. If parsed[key] is defined (including arrays, strings, numbers), use parsed[key]

  This ensures masterPrompt defaults are preserved when the saved file doesn't include them.

  Additionally, as a safety net, update the SettingsSchema in settings.ts to use `.default()` on the masterPrompt field:
  ```typescript
  masterPrompt: MasterPromptSchema.default({ template: DEFAULT_MASTER_PROMPT })
  ```
  But be careful - DEFAULT_MASTER_PROMPT is imported from a main-process file. If this causes circular import issues, keep the default only in the merge logic.

<task type="auto">
  <name>Task 2: Wire settings versioning to SQLite and fix history display</name>
  <files>src/main/services/settings-service.ts, src/main/ipc/settings.ts</files>
  <action>
  **Root cause:** SettingsService.save() writes version backup files to userData/versions/ but NEVER calls `insertSettingsVersion()` from queries.ts. The SettingsHistorySection calls `settingsVersions.list()` which queries the `settings_versions` SQLite table - which is always empty.

  **Fix in settings-service.ts:**
  1. Import `insertSettingsVersion` from `../db/queries`
  2. In the `save()` method, after successfully writing the version file to disk, also record it in SQLite:
     ```typescript
     // After writing version file:
     const timestamp = Date.now()
     const versionFilename = `settings_${timestamp}.json`
     // ... existing file write logic ...

     // Record in SQLite for history UI
     try {
       insertSettingsVersion(1, versionFilename, timestamp)  // brand_id = 1
     } catch (dbErr) {
       console.error('Failed to record settings version in DB:', dbErr)
       // Non-fatal - file version still exists as backup
     }
     ```

  3. The `settings_versions` table schema has columns: id, brand_id, filename, timestamp, version (auto-increment via trigger or manual). Check schema.sql to see if `version` is auto-generated. If not, track version number by querying current max version before insert.

  **Fix in settings IPC:**
  The `settings-versions:list` handler already calls `listSettingsVersions(1)` which is correct. No changes needed to IPC.

  **Fix SettingsHistorySection display:**
  The SettingsHistorySection references `version.version` and `version.filename` and `version.timestamp`. Verify these match the SettingsVersion interface from queries.ts. The `timestamp` in the DB is a number (epoch ms). The component's `formatDate()` creates `new Date(timestamp)` which should work with epoch ms. No changes needed to the component if the data flows correctly.

  **Backfill existing versions (optional but recommended):**
  In `SettingsService.load()`, after loading settings, check if the versions directory has files that aren't in the DB. If so, backfill them. This handles the case where versions were created before this fix:
  ```typescript
  // After loading settings successfully, backfill any unrecorded versions
  try {
    const existingFiles = await this.getVersions()
    const dbVersions = listSettingsVersions(1)
    const dbFilenames = new Set(dbVersions.map(v => v.filename))
    for (const filename of existingFiles) {
      if (!dbFilenames.has(filename)) {
        const ts = parseInt(filename.replace('settings_', '').replace('.json', ''))
        if (!isNaN(ts)) {
          insertSettingsVersion(1, filename, ts)
        }
      }
    }
  } catch (err) {
    console.error('Backfill failed:', err)
  }
  ```
